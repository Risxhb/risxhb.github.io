# Eve: Moonroot Hollow - Current Game Summary

Last reviewed: 2026-05-11

## High-Level Summary

Eve: Moonroot Hollow is a browser-hosted, canvas-native RPG slice. The game runs from `eve/index.html`, draws the full experience inside a single 960x640 canvas, and uses image-generated fantasy assets for characters, environments, menus, title art, tiles, props, enemies, and UI panels.

The current playable fantasy is a small party-based RPG about traveling from Emberleaf Trail into Brindlemarket Village and then descending into Moonroot Hollow. The party is trying to solve the sickness spreading through the roots beneath Brindlemarket by entering the Hollow, opening its deeper backroom path, and recovering the Moonroot Bloom.

The game currently supports:

- A title screen with New Game / Continue and a door-style transition.
- Top-down field exploration with keyboard movement, pointer UI, collision, exits, pickups, NPC interaction, and map transitions.
- A four-member party with visible followers, reorderable party cards, profiles, stats, portraits, and roles.
- Inventory, shops, quest log, settings, save/load, and dungeon map menus.
- Turn-based combat with party turns, enemy turns, skills, items, guarding, fleeing, victory, defeat, XP, gold, and quest progression hooks.
- A generated Moonroot Hollow dungeon system, currently implemented through floor 11, with a future-facing max of 100 floors.
- WebAudio music, ambience, and cue-based sound effects generated in code.
- A data validation tool and several asset-processing / QA scripts.

## Current Player Experience

The player starts from a title screen and can begin a new game. The initial state places the party in `moonroot_hollow`, but the content path also includes an outdoor tutorial map, a village hub, and dungeon floors.

Core controls:

- WASD / arrow keys: move in 8 directions and navigate menus.
- Enter / Space / E: confirm, interact, or advance dialogue.
- Escape / Backspace: cancel or open the field menu.
- I: inventory.
- P: party.
- Q: quest log.
- M: dungeon map.
- Mouse / pointer: menu selection, clicks, sliders, dragging party cards, and wheel zoom / scrolling.

The field loop includes:

- Player movement with collision against blocked tiles, rectangles, NPCs, and closed doors.
- Party followers using a leader-trail system so the group follows the current leader.
- Interactions with NPCs and nearby props.
- Automatic pickup collection for nearby pickup props.
- Tile exits between maps.
- Encounters when stepping onto encounter tiles.
- Dungeon traps and dungeon-region visibility.
- Camera zoom between 0.75x and 2x with the mouse wheel; new games default to the 0.75x zoomed-out view.

## Current Story And Content

The RPG slice revolves around Brindlemarket and Moonroot Hollow.

Main locations:

- `Emberleaf Trail`: outdoor tutorial route with a camp, collectible Emberleaf pickups, Mira dialogue, tutorial encounters, and a locked exit to Brindlemarket until the tutorial is complete.
- `Brindlemarket Village`: hub town with roads, market square, buildings, props, shops, NPCs, one encounter, and exits back to the trail or into Moonroot Hollow.
- `Moonroot Hollow`: dungeon area generated by `eve/src/dungeon.js`, with floor 1 as a bespoke tutorial-style dungeon, floor 2 as a more designed dungeon, and floors 3-11 generated from reusable floor logic.

Party members:

- Rowan Valeguard, Knight / Protector.
- Elara Sunmere, Priest / Healer.
- Cassian Runevale, Mage / Caster.
- Nia Quickwhisker, Thief / Scout.

Enemy data currently includes 11 enemy definitions, including tutorial, field, dungeon, deeper-floor, and boss-role enemies.

Quest data currently includes:

- `A Leaf Still Burning`: tutorial path through Mira, Emberleaf collection, first combat, and reaching Brindlemarket.
- `Market Herbs`: village errand around Emberleaf.
- `Nia's Rooftop Rumor`: village rumor quest.
- `The Root Beneath the Moon`: main Moonroot quest path that leads toward Vorthane and the Moonroot Bloom.

Items currently include healing consumables, MP recovery, food, antidote, revive item, one weapon, one accessory, and key items such as Emberleaf, Moonroot Key, and Moonroot Bloom.

Shops currently include:

- `Luma's Hearth Foods`: bread and stew.
- `Pippa's Provisions`: potion, ether, antidote, revive feather.
- `Tovin's Tools`: iron sword.

## Runtime Architecture

The game is a static browser app with ES modules and JSON data. There is no framework layer. `eve/index.html` hosts the canvas, loads `eve/styles.css`, and imports `eve/src/main.js`.

Primary runtime files:

- `eve/src/main.js`: main game loop, state updates, world interactions, menus, quest scripting, combat, map transitions, dungeon progress, input routing, and save/load calls.
- `eve/src/renderer.js`: all canvas rendering, including title screen, world view, actors, dungeon layers, fog, HUD, dialogue, field menus, inventory, party, profiles, quests, settings, shop, combat, toasts, and transitions.
- `eve/src/state.js`: initial state, localStorage save snapshot, and localStorage load recovery.
- `eve/src/data.js`: JSON loading, generated dungeon map injection, indexing, and runtime data validation.
- `eve/src/dungeon.js`: Moonroot dungeon generation, semantic dungeon tiles, traps, doors, regions, floor progression, encounters, loot, and procedural floor layout.
- `eve/src/input.js`: keyboard and pointer input abstraction.
- `eve/src/assets.js`: image and atlas loading, atlas region lookup, animation frame selection, and draw helper.
- `eve/src/audio.js`: WebAudio graph, procedural music, ambience, audio settings, and UI / combat / item / transition cues.

## Game State Model

The main state object is created by `createInitialState(data)` in `eve/src/state.js`.

Important state sections:

- `mode`: controls the current screen or interaction state, such as `title`, `world`, `dialogue`, `shop`, `map`, `combat`, `menu`, `inventory`, `party`, `quests`, or `settings`.
- `mapId`: current map key.
- `player`: leader position, direction, and movement timer.
- `followers`: follower actor positions and directions.
- `leaderTrail`: cached path samples for follower movement.
- `partyOrder`: actor order, with the first actor as leader.
- `party`: live HP, MP, ultimate meter, level, XP, and status values.
- `flags`: quest, pickup, encounter, trap, door, and story switches.
- `quests`: per-quest stage and completion state.
- `inventory` and `gold`: player resources.
- `doorStates` and `npcStates`: live per-map door / patrol state.
- `dungeon`: current floor, deepest floor, checkpoint info, floor seed, explored regions, current region, and mapped regions.
- `dialogue`, `shop`, `combat`, `menu`, `toast`, `cameraZoom`, `transition`, and `pointerUi`: transient UI and interaction state.

Save/load currently stores a compact gameplay snapshot in localStorage under `eve-rpg-save-v1`. Audio settings are stored separately under `eve-audio-settings-v1`.

## Map And World System

Maps are mostly JSON-driven, with generated Moonroot dungeon maps added at load time.

Map fields used by the runtime include:

- `id`, `name`, `tileSize`, `width`, `height`, and `defaultTile`.
- `fills` and `tiles` for tile painting.
- `tileVariants` for deterministic visual variation.
- `blocked` and `blockedTiles` for collision.
- `props`, `overlays`, `npcs`, `encounters`, and `exits` for world interaction.
- `buildings` for village structures with exterior roofs and interior display regions.
- `rooms`, `corridors`, `regions`, and `regionTiles` for dungeon mapping and fog.
- `dungeonLayers` for semantic dungeon rendering.

World movement works in tile-space with fractional positions. Collision checks block map boundaries, blocked rectangles, blocked tiles, closed doors, and collidable NPCs. Map transitions happen when the rounded current tile matches an exit tile and any required flag is satisfied.

## Dungeon System

Moonroot Hollow is the most developed system in the project.

Current dungeon constants:

- `MAX_DUNGEON_FLOOR = 100`
- `IMPLEMENTED_DUNGEON_FLOOR = 11`
- `DUNGEON_CHECKPOINT_FLOORS = [25, 50, 75]`

Floor generation:

- Floor 1 is a bespoke semantic tutorial dungeon with four rooms, corridors, doors, traps, props, NPCs, and floor 2 descent.
- Floor 2 is a designed dungeon with named rooms, water, traps, encounters, loot, and descent.
- Floors 3-11 are generated from seeded floor logic, with floor 10 acting as a boss floor and floor 11 representing the currently available post-boss deeper content.

Dungeon systems currently include:

- Semantic tile categories such as floor, wall, open doors, water, pit, stairs, chest, and trap tiles.
- Dungeon tile registry mapping semantic concepts to asset IDs.
- Region graphs and reveal links for fog-of-war.
- Doors that open when the party stands nearby.
- Structural fog hiding unexplored dungeon regions.
- Auto-mapping of current and nearby regions.
- Trap types: pitfall, spikes, darts, poison, alarm, crumble, and teleport.
- Pitfalls that can drop the party deeper, bounded by implemented floors and uncleared boss floors.
- Boss-gated descent on floor 10.
- Checkpoint fields in state for future floor 25 / 50 / 75 return anchors.

## Combat System

Combat lives in `eve/src/main.js` and is rendered by `eve/src/renderer.js`.

Combat flow:

- `startCombat(encounter)` creates live enemy instances from actor data.
- Turn order is rebuilt from live party members and enemies, sorted by agility.
- Party turns use the root menu: Attack, Skill, Item, Defend, Flee.
- Skill menus are based on the current actor's configured skill IDs, with ultimate added when the ultimate meter reaches 100.
- Item menus include inventory items usable in battle.
- Enemy turns target the living party member with the lowest HP.
- Victory awards XP and gold, sets the encounter flag, and can advance or complete quest hooks.
- Defeat restores the party to half HP and retreats to the current checkpoint or Brindlemarket.

Combat is currently deterministic and direct. There is no equipment application layer, status duration system, enemy skill AI table, targeting UI, level-up curve, or party member selection UI yet.

## Quest And Script System

NPCs and props call script IDs from `eve/data/npc-scripts.json`.

The script runner supports action kinds such as:

- Dialogue text.
- Setting flags.
- Giving and taking items.
- Starting, advancing, and completing quests.
- Opening shops.
- Starting combat indirectly through encounter logic.
- Transferring maps.

Quest state is stored by quest ID and stage number. Completion rewards XP, gold, and items to the party.

This system is simple but effective for current content. It can already represent tutorial guidance, shops, village conversations, gating flags, and main quest progression.

## Party System

The party currently has four fixed characters. Runtime party state stores live combat values separately from base actor data.

Current party features:

- Visible walking leader and followers.
- Follow spacing through a sampled movement trail.
- Menu command to rotate the leader.
- Pointer drag-and-drop party card reorder.
- Party cards with HP / MP / ultimate meter information.
- Character profile screens with portraits, biography text, and stats.
- Actor-specific skill lists and ultimate skills.

The first party member in `partyOrder` is the field leader. Reordering rebuilds followers and can change the displayed leader.

## Inventory, Shops, And Economy

Inventory is a simple ID-to-count object. Items define price, sell value, icon, usability contexts, and effects.

Implemented item effects:

- Single-target healing.
- Party healing.
- MP restoration.
- Revive.
- Cleanse metadata exists, though the current field item handler only applies a subset directly.

Shops are JSON-driven and opened through scripts. Buying checks gold, subtracts price, adds one item, plays a cue, and shows a toast.

Gear exists in item data but there is not yet an equipment UI or stat-mod application system in active gameplay.

## Rendering System

Rendering is fully canvas-based. The HTML page only provides the canvas shell.

Renderer responsibilities include:

- Title backdrop scenes, title party art, title menu, and door transition.
- Tile maps, fills, variants, props, overlays, buildings, NPCs, player, followers, encounters, and sorted draw order.
- Dungeon-specific layered rendering and fog-of-war.
- HUD with party and location information.
- Dialogue panels.
- Menus for field menu, inventory, party, quests, settings, shop, combat, and dungeon map.
- Party profile layouts and scrollable biography text.
- Toast messages and transition effects.

Assets come from a manifest with 2 atlases, 78 atlas regions, 214 image entries, and 4 NPC sprite mappings at the time of this review.

## Audio System

Audio is generated in `eve/src/audio.js` with WebAudio.

Current audio features:

- User-gesture unlock.
- Master, music, SFX, and ambience gain controls.
- Saved audio settings.
- Procedural field, tutorial forest, combat, and boss music scheduling.
- Wind and water ambience.
- UI, item, attack, skill, enemy, guard, encounter, victory, transition, and wake cues.
- Combat pulse layer for combat / boss scenes.

The runtime audio manager is procedural and does not require a bundled music file.

## Current Boundaries And Likely Next Systems

The project is already a playable vertical-slice RPG shell, but several systems are still early or placeholder-level.

Current boundaries:

- Dungeon floor generation is implemented through floor 11, even though the design constants point toward 100 floors.
- Checkpoints are represented in state but future floor 25 / 50 / 75 checkpoint behavior is not fully realized in playable content yet.
- Gear items exist, but equipping gear and applying stat modifiers is not implemented as a full gameplay loop.
- Skills have data-rich identities, but combat resolution currently handles a limited set of effects.
- Status effects exist as state fields but do not yet form a complete status lifecycle.
- Enemy behavior is simple and does not yet use per-enemy skill AI.
- Quest and script systems are useful but intentionally lightweight.
- Save/load persists core gameplay state but not every transient runtime detail.
- The root README describes the sprite-generation scaffold, while `eve/README.md` describes the actual game.

Likely next high-value systems:

- Equipment UI and stat application.
- Level-up / progression curve.
- Expanded enemy skill AI and target selection.
- Full status effect lifecycle.
- More dungeon floor content beyond floor 11.
- Checkpoint unlock and return flow.
- More robust quest journal details and objective display.
- More complete field item effect handling.
- Better separation of large `main.js` responsibilities into world, combat, menu, and script modules when the codebase grows further.

## Near-Term Design Direction

The next major push should treat the dungeon as its own responsibility area, not just one part of `main.js`. The goal is to get Moonroot Hollow's dungeon system built enough that the remaining floors can be produced, themed, validated, and visually tuned without rewriting core gameplay every time.

Detailed next-burn biome, tileset, sprite, and dungeon-survival guidance now lives in `NEXT_BURN_DUNGEON_BIOME_TILESET_BRIEF.md`.

The most important near-term dungeon work is:

- Rework floors 1-10 into the first coherent dungeon arc, with floor 1 serving as an Adventurer's Guild outpost / forward base inside Moonroot Hollow.
- Separate dungeon generation, dungeon rendering data, trap placement, floor progression, and biome/theming rules into clearer modules.
- Make the current floors a template for building the rest of the dungeon, including room/corridor patterns, boss gates, checkpoints, hazards, loot, encounters, and floor-to-floor transitions.
- Improve the look of dungeon floors using the textures and sprite sets already in the project.
- Add a clean way to change level themes, so new biome art from sprite generation can plug into the dungeon system without hardcoding every tile by hand.
- Keep visual QA central: dungeon floors need screenshots and regression checks for walls, doors, fog, traps, biome tiles, and scale.

Combat is still a major future system, especially because skills, lore, character background, enemy identity, and battle feel all need another pass. But the next productive chunk is likely dungeon infrastructure first, then combat once the dungeon can reliably produce places worth fighting through.

## Future Loot And Dungeon Survival System

A major future system idea is detailed monster loot and dungeon-based survival. Instead of enemies only dropping gold or generic items, monsters can produce loot based on what they are, how they died, and what condition their body and gear are in.

Example direction:

- A goblin might have ears, worn gear, tattered clothes, weapons, carried items, food scraps, and meat.
- Body-part loot can depend on damage type. Fire might ruin ears or meat. Slashing might damage hides, ears, or gear. Blunt damage might preserve skins but break bones or equipment.
- Gear can have condition states such as broken, worn, tattered, bloodied, burned, cracked, usable, or repairable.
- Drops should distinguish between biological materials, equipment, carried inventory, quest trophies, cooking ingredients, and salvage.
- The dungeon should support a "living off the dungeon" loop, where deeper exploration is possible because the party can harvest, cook, preserve, or prepare what they find instead of always returning to town.

This points toward a deeper dungeon ecology system:

- Monsters should have anatomy / material profiles.
- Attacks and skills should carry damage tags that affect loot outcomes.
- Cooking or camp systems can turn monster parts, herbs, fungi, water, and supplies into meals.
- Meals can restore HP/MP, grant temporary buffs, reduce hunger/fatigue, cure conditions, or create risks if prepared badly.
- Some ingredients may be culturally taboo, poisonous, cursed, unstable, or only useful after preparation.
- Town resupply remains useful, but skilled dungeon survival should make long expeditions possible.

This system should eventually connect dungeon design, combat, inventory, crafting/cooking, lore, and party personality. It can start small with a few enemies and a simple loot-condition table, then grow into a signature mechanic.
